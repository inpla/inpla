# Visualising Parallel Rewriting with `parallel_analysis.plt`

This document explains how to use the gnuplot script `parallel_analysis.plt` stored under the `tools/` directory to visualise execution profiles produced by Inpla in parallel execution simulation mode.

The script is designed to plot:

- the number of **executed active pairs** (left y-axis), and
- the number of **existing agents** (right y-axis)

as a function of execution steps, based on CSV output generated by Inpla.

---

## 1. Prerequisites

You will need:

- **Inpla** (with parallel execution simulation mode enabled)
- **gnuplot** (a version supporting `cairo` terminals is recommended)
- CSV data generated by Inpla using the `-s` option

The script expects CSV files with a header of the following form:

```text
step,exec(0),stacked,agents,rule1,rule2,...
```

Here:

- `step` is the execution step,
- `exec(0)` is the number of active pairs executed at that step,
- `stacked` is the number of active pairs remaining after the execution,
- `agents` is the number of agents currently present,
- `rule1`, `rule2`, ... are the numbers of active pairs executed by each rewrite rule.

---

## 2. Recommended Directory Layout

A simple directory structure is:

```text
project/
├── inpla
├── add1.in
├── add2.in
├── tmp.csv
└── tools/
    ├── parallel_analysis.plt
    └── parallel_analysis.md
```

All command examples below assume that you run them from the project root directory.

---

## 3. Basic Usage (Default Settings)

If your CSV file is named `tmp.csv`, simply run:

```bash
gnuplot tools/parallel_analysis.plt
```

This generates a PDF file named:

```text
analysis.pdf
```

using default settings.

---

## 4. Customising Input and Output Files

You can specify the input CSV file and output filename using gnuplot’s `-e` option:

```bash
gnuplot -e "file='my_data.csv'; outfile='result.png'" tools/parallel_analysis.plt
```

Supported output formats include:

- `.pdf` (default)
- `.png`
- `.eps`

The output format is automatically detected from the file extension.

Examples:

```bash
gnuplot -e "file='run1.csv'; outfile='run1.pdf'" tools/parallel_analysis.plt
gnuplot -e "file='run2.csv'; outfile='run2.png'" tools/parallel_analysis.plt
gnuplot -e "file='run3.csv'; outfile='run3.eps'" tools/parallel_analysis.plt
```

---

## 5. Adding a Custom Title

You can change the plot title dynamically by passing the variable `title_text`:

```bash
gnuplot -e "file='my_data.csv'; outfile='result.eps'; title_text='Simulation Run A'" tools/parallel_analysis.plt
```

The variable name `title_text` is used instead of `title` to avoid clashes with gnuplot’s internal `set title` command.

Example:

```bash
gnuplot -e "file='tmp.csv'; outfile='assoc1.eps'; title_text='(a+b)+c'" tools/parallel_analysis.plt
```

---

## 6. Adjusting Size for LaTeX (EPS Output)

When generating EPS figures for LaTeX documents, you can control the figure size and font using:

- `eps_w` : width in inches
- `eps_h` : height in inches
- `eps_fsize` : base font size

Example:

```bash
gnuplot -e "file='tmp.csv'; outfile='graph1.eps'; eps_w=4.5; eps_h=3.5; eps_fsize=12" tools/parallel_analysis.plt
```

This is useful when placing multiple figures side by side in a single column.

Notes:

- These parameters are mainly intended for **EPS output**.
- For `.pdf` and `.png`, the script uses built-in default sizes.

---

## 7. Setting Axis Limits and Tick Steps

For precise comparison between multiple runs, axis ranges and tick intervals can be fixed manually.

### X-axis (execution steps)

- `x_max` : maximum step shown

### Y-axis (executions, left axis)

- `y_max` : maximum value
- `y_step` : tick interval

### Y2-axis (agents, right axis)

- `y2_max` : maximum value
- `y2_step` : tick interval

Example:

```bash
gnuplot -e "file='data1.csv'; x_max=150; y_max=3000; y_step=500; y2_max=200; y2_step=50" tools/parallel_analysis.plt
```

If limits are not specified, suitable ranges are automatically inferred from the data.

Another example:

```bash
gnuplot -e "file='tmp.csv'; outfile='custom.pdf'; title_text='Custom Range'; x_max=100; y_max=20; y_step=5; y2_max=200; y2_step=25" tools/parallel_analysis.plt
```

---

## 8. Plot Contents

The generated plot consists of:

- **Stacked bar charts** on the left y-axis  
  These show the number of active pairs executed at each step, broken down by rewrite rules (`rule1`, `rule2`, ...).

- **A line plot** on the right y-axis  
  This shows the total number of agents present in the net at each step.

In other words:

- the **bars** represent execution activity,
- the **line** represents the size of the net.

Additional features include:

- automatic thinning of x-axis ticks for long executions,
- optional vertical marker if execution ends before the specified `x_max`,
- consistent colour cycling for large numbers of rewrite rules.

---

## 9. Typical Workflow with Inpla

A typical analysis workflow is as follows.

### Step 1: Run Inpla in parallel execution simulation mode

Inpla writes CSV data to `STDERR`, so you should redirect it to a file:

```bash
./inpla -f add1.in -s 0 2> tmp.csv
```

Here:

- `-f add1.in` specifies the input file,
- `-s 0` runs Inpla in parallel execution simulation mode with unlimited threads,
- `2> tmp.csv` redirects `STDERR` to `tmp.csv`.

### Step 2: Visualise the CSV file

```bash
gnuplot -e "file='tmp.csv'; outfile='add1.eps'; title_text='(a+b)+c'" tools/parallel_analysis.plt
```

This produces a plot of the execution profile stored in `tmp.csv`.

## 11. Automatic Behaviour of the Script

The script automatically performs the following tasks:

1. reads the CSV file,
2. detects the number of columns,
3. determines the number of rows,
4. computes a suitable range for the right y-axis (`agents`),
5. adjusts x-axis tick density depending on the number of steps,
6. chooses the output terminal based on the extension of `outfile`,
7. plots rule-by-rule execution counts as stacked histograms,
8. overlays the number of agents as a line plot.

This means that, in most cases, you only need to specify:

- `file`
- `outfile`
- optionally `title_text`

and the rest works automatically.

---

## 12. Console Output

After plotting, the script prints a short summary to the console, including:

- the effective x-axis maximum,
- the y-axis maximum for executions,
- the y2-axis maximum for agents.

This information is useful for quick sanity checks and for tuning axis parameters.

A typical console output looks like this:

```text
=========================================
 Graph plotted successfully:  analysis.pdf
-----------------------------------------
 X-axis max (step)   : ...
 Y-axis max (Exec)   : ...
 Y2-axis max (Agents): ...
=========================================
```

---

## 13. Intended Use

The script `parallel_analysis.plt` is intended as a lightweight visualisation and analysis tool to:

- understand the degree of parallelism in interaction net programs,
- compare different formulations of the same computation,
- identify sequential bottlenecks and parallel-friendly structures,
- support experimental workflows in papers, demos, and implementation studies.

It is especially useful when comparing different net structures that compute the same result but expose different amounts of parallelism.

---

## 14. Summary

The basic pattern is:

1. run Inpla and redirect `STDERR` to a CSV file,
2. run `tools/parallel_analysis.plt` on the CSV file,
3. inspect the resulting graph.

Minimal example:

```bash
./inpla -f add1.in -s 0 2> tmp.csv
gnuplot -e "file='tmp.csv'; outfile='add1.pdf'; title_text='(a+b)+c'" tools/parallel_analysis.plt
```

This provides a simple and reusable workflow for analysing the dynamic behaviour of parallel rewriting in interaction nets.
